# 🔄 Engineering PDCA — 工程实践方法论

## 核心定位

将 PDCA 循环（Plan → Do → Check → Act）作为项目的**心跳节拍**，在每个循环中注入四个哲学原则，确保工程实践既有纪律又有弹性。

本技能不替代任何具体技能，而是提供**节奏感**和**判断框架**——告诉你「现在该干什么」「干得怎么样」「下一步往哪走」。

---

## 四大哲学原则

### 1. 容错辩证法（Fault-tolerant Dialectics）

> 错误不是失败，是信息量最高的知识生产形式。

**核心信条**：
- **快速失败 > 延迟成功**：一个 30 分钟的试错比 3 天的「完美方案」更有价值
- **每次错误必须产出知识**：错误发生后，必须在 SSOT 中记录「为什么错 + 怎么发现 + 下次怎么避免」
- **容错有边界**：每次试错前声明边界（时间上限、影响范围、回滚条件），越界立即止损

**反模式**：
- 「这个方案不够完美，我再想想」→ 想 3 天不如试 30 分钟
- 「刚才那个错误太蠢了，别提了」→ 不提 = 白错
- 「先不动，看看会不会自己好」→ 工程问题不会自愈

### 2. 主次矛盾（Primary/Secondary Contradictions）

> 任何一个时刻，有且只有一个主要矛盾。抓住它，其他暂时放下。

**核心信条**：
- 每个 PDCA 周期的 **P 阶段必须明确：「当前周期的主要矛盾是什么？」**
- 次要矛盾记录在案（SSOT 的 `PARKING_LOT.md`），不遗忘、不纠缠
- 主要矛盾解决后，重新评估：哪个次要矛盾上升为主要矛盾？
- 如果发现自己在同时追 3 个问题 → 你还没找到主要矛盾

**判断主要矛盾的方法**：
1. 列出当前所有待解决的问题
2. 对每个问题问：「解决它之后，其他问题是否变简单了？」
3. 回答「是」最多的那个 → 主要矛盾

### 3. 实践标准（Practice Standards）

> 标准不是写在文档里的，是能跑通、能回滚、能交接的。

**三级验收标准**：

| 级别 | 标准 | 验证方式 |
|------|------|----------|
| L1 能跑通 | 功能在本地/测试环境正常运行 | `pytest` 全绿 + 手工冒烟 |
| L2 能回滚 | 出问题可以一键退回上一个稳定状态 | `git revert` + `make sync` |
| L3 能交接 | 另一个人（或 3 天后的自己）能看懂并继续 | SSOT 文档已更新 + `make feat` 分支干净 |

**反模式**：
- 「差不多了，先上线看看」→ 连 L1 都没过
- 「回滚方案？main 分支应该没问题吧」→ L2 缺失
- 「这块只有我知道怎么改」→ L3 缺失 = 巴士因子 = 1

### 4. SSOT 的动态性（Dynamic Single Source of Truth）

> SSOT 不是写完就供起来的碑文，是每次 Check 后都要更新的活文档。

**SSOT 文档体系**：

```
项目根目录/
├── README.md              # 项目门面：做什么、怎么跑、当前版本
├── CLAUDE.md              # AI 上下文：架构决策、当前状态、坑位地图
├── ROADMAP.md             # 路线图：已完成 / 进行中 / 规划中
├── OPERATIONS_MANUAL.md   # 运维手册：部署、监控、故障处理
└── PARKING_LOT.md         # 暂存区：次要矛盾、待评估想法、技术债
```

**动态更新规则**：
- **D 阶段每完成一个任务** → 更新 CLAUDE.md 的「当前状态」
- **C 阶段发现任何 gap** → 立即更新相关 SSOT 文档
- **A 阶段发布完成后** → 更新 README.md + ROADMAP.md + auto-doc-update 全量同步
- **PARKING_LOT 的清理纪律**：每个 PDCA 周期至少认领或删除 1 条暂存项

## PDCA 工作流

### P — Plan（计划）

**目标**：明确「这轮做什么、不做什么、为什么」。

**输入**：
- 上一轮 A 阶段产出的 SSOT
- PARKING_LOT.md 中的待评估项
- 用户/业务的最新需求

**执行步骤**：

```
1. 【主次矛盾分析】
   - 列出当前所有待解决问题
   - 用「解决它后其他问题是否变简单」筛选出主要矛盾
   - 在 CLAUDE.md 中写入：## 当前PDCA周期 / 主要矛盾: {一句话描述}

2. 【范围界定】
   - 明确本轮做什么（In Scope）
   - 明确本轮不做什么（Out of Scope → 写入 PARKING_LOT.md）
   - 设定时间盒（Time Box）：本轮预计多久？超时怎么办？

3. 【SSOT 基线确认】
   - 确认 README.md、CLAUDE.md 反映了当前真实状态
   - 如不一致 → 先修复 SSOT 再开始新周期

4. 【分支创建】
   - 调用 git-workflow: make feat NAME=<本轮核心特性>
```

**产出物**：
- ✅ CLAUDE.md 中写明了当前周期的主要矛盾
- ✅ In Scope / Out of Scope 清单
- ✅ 时间盒设定
- ✅ 专用 feature 分支已创建

### D — Do（执行）

**目标**：在容错边界内快速迭代，每个任务产出可验证的结果。

**执行原则**：

```
1. 【小步快跑】
   - 每个任务拆分到 ≤ 30 分钟可完成
   - 每完成一个任务 → git commit（原子提交）
   - 每完成一个任务 → 更新 CLAUDE.md 当前状态

2. 【容错边界声明】（任务粒度 > 30 分钟时必须）
   - 时间上限：最多试多久？
   - 影响范围：改动涉及哪些文件/模块？
   - 回滚条件：什么信号出现时立即放弃？
   - 预期产出：即使失败了，能学到什么？

3. 【错误即知识】
   - 每次试错失败 → 在 CLAUDE.md 的「坑位地图」区记录：
     - 尝试了什么
     - 为什么失败了
     - 下次怎么避免
   - 格式：## 坑位地图 / {日期} {简述} → {根因} → {教训}

4. 【SSOT 随动更新】
   - 任何架构决策变更 → 立即更新 CLAUDE.md
   - 任何接口/配置变更 → 立即更新 OPERATIONS_MANUAL.md
```

**产出物**：
- ✅ 每个任务有原子 commit
- ✅ 坑位地图持续更新
- ✅ CLAUDE.md 保持「当前状态」最新

### C — Check（检查）

**目标**：用实践标准验证产出，用主次矛盾视角重新审视。

**执行步骤**：

```
1. 【L1 验证 — 能跑通】
   - 运行全量测试: pytest tests/ -v
   - 手工冒烟：核心流程走一遍
   - 结果：通过 / 未通过（记录失败原因）

2. 【L2 验证 — 能回滚】
   - 确认 git log 干净、commit 原子化
   - 确认可以 git revert 到本周期开始前的状态
   - 确认 make sync 不产生冲突

3. 【L3 验证 — 能交接】
   - 检查 CLAUDE.md：3 天后的自己能看懂吗？
   - 检查 README.md / OPERATIONS_MANUAL.md：信息是否落后于代码？
   - 触发 auto-doc-update 做文档一致性审计

4. 【主次矛盾回顾】
   - 本轮的主要矛盾解决了吗？
   - 如果没有 → 为什么？需要调整策略还是调整矛盾定义？
   - 如果解决了 → 哪个次要矛盾上升为主要矛盾？
   - 更新 CLAUDE.md 中的「当前PDCA周期」状态

5. 【SSOT Gap 扫描】
   - 代码中有没有文档没说清楚的东西？
   - 文档中有没有代码已经不存在的东西？
   - 发现 gap → 立即修复
```

**产出物**：
- ✅ L1/L2/L3 三级验证结果
- ✅ 主次矛盾状态更新
- ✅ SSOT gap 已修复

### A — Act（改进）

**目标**：把验证过的产出固化为基线，把经验沉淀为知识，准备下一轮循环。

**执行步骤**：

```
1. 【知识沉淀】
   - 将本轮「坑位地图」中有价值的教训提炼为通用规则
   - 更新 OPERATIONS_MANUAL.md 的 FAQ / 故障处理章节
   - 如果某条规则有普适性 → 考虑沉淀到本技能本身

2. 【SSOT 终态同步】
   - 更新 README.md：版本 Badge、Roadmap 进度
   - 更新 ROADMAP.md：标记已完成项、新增规划项
   - 更新 CLAUDE.md：关闭当前周期、标记新的主要矛盾（如有）

3. 【发布交付】（如本轮产出需要发布）
   - 触发 release-superpowers-v2 流水线
   - 或手动执行: make publish V=x.y.z

4. 【清理与复盘】
   - 合并 feature → release → main
   - 删除已合并的 feature 分支
   - 清理 PARKING_LOT.md：认领或删除过期暂存项

5. 【下一轮预告】
   - 在 CLAUDE.md 中写入下一轮的主要矛盾（候选）
   - 设定下一轮的时间盒建议
```

**产出物**：
- ✅ SSOT 全量同步完成
- ✅ 版本发布（如需要）
- ✅ 分支清理
- ✅ 下一轮主要矛盾（候选）

## 触发规则（Trigger Patterns）

| 你说的话 | 技能行为 |
|----------|----------|
| 「开始新项目」「初始化项目」 | 执行 P 阶段：SSOT 骨架 + 主次矛盾分析 + 创建分支 |
| 「开始新的一轮」「下一轮做什么」 | 执行 P 阶段（基于上一轮 A 的产出） |
| 「这个阶段做完了」「检查一下」 | 执行 C 阶段：L1/L2/L3 全量验证 |
| 「发布」「上线」 | 执行 A 阶段：知识沉淀 + 发布交付 |
| 「复盘一下」「回顾一下最近的工作」 | 执行 C→A 完整回顾 |
| 「现在卡住了」「下一步怎么办」 | 触发主次矛盾重新分析 |
| 「这个错误/坑记录一下」 | 触发「错误即知识」→ 写入坑位地图 |

```markdown

**周期编号**: PDCA-{序号}
**启动日期**: {YYYY-MM-DD}
**时间盒**: {N} 天 / 到 {YYYY-MM-DD}

### 主要矛盾
{一句话描述当前最需要解决的问题}

### 范围
- **In Scope**:
  1. {具体任务}
  2. ...
- **Out of Scope** (→ PARKING_LOT.md):
  1. {暂缓任务}
  2. ...

### 验收标准
- L1 能跑通: {具体可验证的条件}
- L2 能回滚: {回滚方案}
- L3 能交接: {需要更新的文档}
```

```markdown

**完成日期**: {YYYY-MM-DD}

### 主要矛盾状态

- [ ] 未解决 — {根因分析 + 调整策略}

### 三级验证
- L1 能跑通: {通过/未通过} — {测试结果}

### 下一轮主要矛盾（候选）
{从 PARKING_LOT.md 或本轮发现中提取}
```

## Gotchas

**PDCA 不是水多了加面**
- 每个周期必须有明确的「完成」定义，避免无限循环
- 如果一个周期超过时间盒 2 倍还没结束 → 强制复盘，重新规划

**容错不等于无底线**
- 每次试错前声明边界，声明了就遵守
- 「再试一次说不定就行了」→ 这是赌博，不是工程

**主次矛盾会骗人**
- 你以为是性能问题，其实是架构问题
- 你以为要做新功能，其实要修旧 bug
- 判断标准不是直觉，是「解决它之后其他问题是否变简单」

**SSOT 的最大敌人是「等会儿再更新」**
- 代码改完 5 分钟后，你对改动的记忆已经开始衰减
- 文档更新必须在同一 PDCA 周期内完成，不能「下个周期补」
- 累积 3 个周期的文档债 → 项目已不可交接
